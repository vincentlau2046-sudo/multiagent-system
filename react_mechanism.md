# React机制实现方案

## 1. 自动重试机制

### 触发条件
- 置信度 < 70%
- Agent执行过程中出现可恢复错误
- 超时但未完全失败

### 重试策略
- **最大重试次数**: 3次（可配置）
- **退避算法**: 指数退避 + 随机抖动
  - 第1次重试: 立即重试
  - 第2次重试: 延迟 2^1 + random(0,1) 秒  
  - 第3次重试: 延迟 2^2 + random(0,2) 秒
- **重试范围**: 
  - 失败的单个Agent重新执行
  - 如果主Agent失败，整个任务重新调度

### 重试状态跟踪
```json
{
  "retryCount": 0,
  "maxRetries": 3,
  "lastRetryTime": "timestamp",
  "retryReason": "置信度不足|执行错误|超时"
}
```

## 2. 任务拆分机制

### 触发条件  
- 任务复杂度评分 > 80（基于关键词数量、依赖关系复杂度）
- 单次执行时间预估 > 10分钟
- 置信度在60-69之间（接近通过但不够）

### 拆分策略
- **按维度拆分**: 将任务按质量评估维度拆分为子任务
  - 完整性子任务: 确保覆盖所有要素
  - 准确性子任务: 专门验证关键信息准确性  
  - 深度子任务: 专注提供深度分析
  - 实用性子任务: 优化可操作性建议

- **按Agent能力拆分**: 根据Agent专长分配子任务
  - ai_intel: 负责情报收集和数据验证
  - tech_analyst: 负责技术分析和架构评估  
  - content_creator: 负责内容优化和实用建议

### 子任务协调
- 主Agent负责子任务调度和结果整合
- 子任务间支持依赖关系定义
- 支持并行和串行混合执行模式

## 3. 人工介入机制

### 触发条件
- 重试3次后仍然失败
- 置信度 < 50% 且涉及关键业务
- 用户明确要求人工审核

### 介入流程
1. **自动暂停**: 停止当前自动执行流程
2. **生成审核包**: 包含当前状态、失败原因、已执行步骤
3. **通知用户**: 通过飞书/Telegram发送审核请求
4. **等待决策**: 
   - 用户确认继续自动执行
   - 用户提供修正指导
   - 用户手动完成剩余工作
5. **恢复执行**: 根据用户决策继续或终止

### 审核包内容
```json
{
  "taskId": "任务ID",
  "currentStatus": "当前状态",
  "failureReason": "失败原因分析", 
  "executedSteps": ["已完成的步骤"],
  "pendingSteps": ["待完成的步骤"],
  "qualityScores": {"完整性": 60, "准确性": 45, "深度": 70, "实用性": 55},
  "suggestedActions": ["重试", "拆分任务", "人工修正", "终止任务"]
}
```

## 4. 降级输出机制（需确认）

### 触发条件
- 用户明确启用降级输出
- 任务超时但已有部分结果
- 资源限制导致无法完成完整任务

### 降级策略
- **时间降级**: 在指定时间内提供简化版本
- **质量降级**: 降低某些维度要求（如深度、实用性）
- **范围降级**: 缩小任务覆盖范围，聚焦核心要素

### 确认流程
⚠️ **重要**: 降级输出必须经过用户确认，避免token浪费

1. 检测到需要降级的情况
2. 生成降级方案预览（包含预计token消耗）
3. 向用户发送确认请求
4. 用户确认后才执行降级输出
5. 记录降级决策用于后续优化

### 降级确认消息模板
```
【降级输出确认】
任务: {任务名称}
当前状态: {置信度}% ({失败原因})
降级方案: {降级策略描述}
预计消耗: {token数量} tokens
是否确认执行降级输出？[是/否]
```
